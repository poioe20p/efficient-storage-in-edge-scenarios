=== Theme: Multi-Controller Anycast & VIP Coordination ===

Topic: Implementation of a Distributed Anycast Load Balancing Framework in Multi-Controller SDN Environments.
- Focus: The mechanics of the SDN-Anycast. How do you manage a single Virtual IP across three disjoint L2 domains?
- Research Question: How can isolated controllers coordinate to provide a seamless Anycast service for a database cluster without
 a central network view?
- MongoDB Role: Acts as a basic heartbeat registry.
- NetworkX Role: Calculates local "shortest path" to the local instance vs. the "path to the router."
- Thesis Contribution: A functional proof-of-concept of SDN-based server-side load balancing that replaces
 expensive hardware balancers.


=== Theme: Data Locality & Shard-Aware Routing ===

Topic: Metadata-Aware Load Balancing for Sharded NoSQL Databases in Federated SDN-Edge Networks.
- Focus: Avoiding the "Internal Jump" (East-West traffic).
- The Problem: Standard Anycast might send a request to Network A, but the MongoDB data lives in Network B.
 This forces the database to fetch the data across the router anyway.
- Research Question: Can the SDN controller reduce total query latency by inspecting 
 application-layer shard keys and routing traffic directly to the "Data-Home" shard?
- MongoDB Role: The controller queries the config database to map Shard Key Ranges \(\rightarrow\) Physical Networks.
- Novelty: You are moving from "Closest Server" to "Closest Server with the Correct Data."
- Metric: Measure the reduction in "Inter-shard chunk migration" and router traffic.


=== Theme: Multi-Metric Load Steering (Network + Database) ===

Topic: Joint Optimization of Network Topology and Database Metrics for Dynamic Load Steering at the Edge.
- Focus: Using MongoDB performance metadata to influence SDN paths.
- The Problem: A server is close (1 hop) but its disk I/O is saturated.
 Another server is far (3 hops) but is idle.
- Research Question: What is the optimal weighted function between Network Latency (RTT)
 and Database Execution Time for a distributed NoSQL workload?
- MongoDB Role: Provide real-time metrics (opcounters, mem.resident, wiredTiger.cache).
- NetworkX Role: Calculate path costs that are weighted by server load.
- Thesis Contribution: A "Composite Cost Function" (\(C\)) that balances network congestion against server exhaustion.
 You prove that looking at the DB state is better than just looking at the Switch state.


=== Theme: Proactive Control Loops (Prediction + Elasticity) ===

Topic: Predictive Resource Allocation for Edge Computing based on Spatio-Temporal Data Popularity Trends.
- Focus: Using historical logs to anticipate "Hotspots."
- The Problem: Re-routing is reactive. By the time you detect a load spike, the user has already experienced lag.
- Research Question: Can historical access patterns stored in MongoDB be used to proactively "Warm up" 
 SDN paths and Anycast weights before a peak occurs?
- MongoDB Role: Use Time-Series collections to store the logs of where/when requests came from.
- Thesis Contribution: Using a "Moving Average" or "Trend Analysis" model stored in the database to drive 
 SDN "Group Table" weight updates.
- Metric: Measure "Jitter" and the speed of adaptation to sudden traffic bursts.


Topic: "Traffic-Pressure" Scaling (Proactive vs. Reactive) -> python.subprocess(bash_script)
- The Idea: Standard scaling (like Kubernetes HPA) waits for the Server CPU to hit 80%. By then, the database is already sluggish.
- The Research: The SDN controller monitors the Bitrate/Packet-rate on the switch ports. When it sees a sudden surge in traffic 
 destined for the MongoDB Anycast VIP, it triggers a docker run to spin up a new MongoDB Secondary before the existing server's
 CPU spikes.
- Thesis Contribution: Proving that 
 "Network-Layer Signaling" leads to faster scaling and fewer "dropped" queries than "Application-Layer Monitoring."


Topic: "Energy-Aware" Consolidation (Scale-Down) -> python.subprocess(bash_script)
- The Idea: In Edge computing, power is expensive.
- The Research: When the SDN controller sees the total query rate drop below a threshold (e.g., late at night), it:
	1. Uses pymongo to ensure all data is safely replicated to a "Master" node.
	2. Removes the extra nodes from the MongoDB replica set.
	3. Shuts down the Docker containers to save power.
	4. Updates the SDN "Anycast" flow rules to point everything to the single active node.
- Thesis Contribution: Measuring the trade-off between "Power Consumption" and "Burst Readiness."


=== Theme: Adaptive Data Management (Reliability, Storage, Computation, Placement) ===

Topic: Programmable Replication Factors based on Network Health
Standard MongoDB has a fixed replication factor (e.g., 3 copies). In Edge computing, bandwidth is scarce.
- The Idea: The SDN controller monitors the available bandwidth between your 3 networks.
- The Research: When the SDN detects high network congestion (via OVS stats), it "programs" the MongoDB container to reduce its replication factor (e.g., from 3 to 2) to save bandwidth. When the network is idle,
 it triggers a scale-up of replicas to improve data safety.
- Thesis Contribution: Adaptive Reliability. Trading off data redundancy for network performance dynamically.


Topic: SDN-Driven "Data Thinning" at the Edge
Edge containers often run out of storage space. Not all data is equally important.
- The Idea: Use the SDN controller to identify "Cold Data" (metadata/collections that haven't been routed to in 24 hours).
- The Research: The SDN controller triggers a programmable action inside the container to compress, downsample,
 or move that data to a "Cold Shard" in a different network (or the Cloud) to free up high-speed SSD space for "Hot Data."
- Thesis Contribution: Hierarchical Edge Storage. Using the SDN controller as the "Data Life Cycle" manager.


Topic: Containerized "Data-Local" Query Offloading
Instead of moving massive amounts of data to a central server to be processed, you move the query container to the data.
- The Idea: When a heavy aggregation query is detected, the SDN controller decides where to execute it.
- The Research: Based on the NetworkX topology, the controller calculates if it is cheaper to move the Data to the Query
 or spin up a New MongoDB Container (Programmable Container) on the switch where the data currently resides.
- Thesis Contribution: Computational Storage. Minimizing the "Data-to-CPU" distance in edge environments.


Topic: Fragmented Sharding for Energy-Efficient Storage
- The Idea: Each of your 3 networks has a different power/resource cost.
- The Research: The SDN controller "programs" the MongoDB sharding logic to move "Popular" chunks to the most
 energy-efficient network.
- Implementation: Using pymongo to call sh.moveChunk() based on energy-weighted paths calculated in NetworkX.
- Thesis Contribution: Green Data Placement. Proving that SDN can make a database cluster "Energy-Aware."
