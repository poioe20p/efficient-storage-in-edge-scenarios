**p95 exceeds 10s for Poll-30s as required** (10.1–10.3s). However, the latency distribution is nearly identical across ALL modes — p50 ~0.1s, p95 ~9.5–10.3s, p99 ~10.3–10.6s. This strongly suggests that **OK-latency data is still effectively censored at ~10.5s**, despite `curl --max-time=30s`. The p99 values being only 0.1–0.3s above p95 across all modes indicates a hard ceiling, not a natural tail.

The plan expected Poll-30s+curl=30s to "reveal the pathological storage_storm saturation regime" with p95 well above 10s. This was not observed — the latency ceiling appears uniform across modes at ~10.5s regardless of polling interval.

**Possible root cause**: a server-side timeout (MongoDB `maxTimeMS`, Flask/gunicorn timeout, or OS-level socket timeout) at ~10s that kills slow requests before they reach the 30s curl limit. This would explain why raising `curl --max-time` from 10s to 30s produced no latency tail elongation compared to v2-lite.

## Overall Verdict

**The experiment completed successfully (12/12 runs, 0 tracebacks) but did not produce the clean monotonic dataset the plan anticipated.** The dominant finding is extreme within-mode variance in failure rates, with each mode showing both "healthy" (~1–2%) and "degraded" (13–61%) runs. This bimodal behavior obscures any between-mode trend and makes the dataset unsuitable for the simple per-mode comparison the plan's hypothesis requires.

The latency distribution further complicates the picture: despite `curl --max-time=30s`, OK-latency is effectively censored at ~10.5s across ALL modes and ALL runs. Raising the client timeout did not reveal the expected tail elongation for Poll-30s.

### Root Cause Hypotheses (ranked)

1. **Bimodal system state** (highest confidence): The system appears to have two stable regimes — "healthy" and "degraded." The trigger for transition between them is not telemetry mode. Likely candidates: Docker resource contention, MongoDB connection pool exhaustion, or a race condition in the VIP routing layer that occasionally routes requests suboptimally, cascading into widespread failures.

2. **Server-side timeout at ~10s** (high confidence): A component in the request path (MongoDB `maxTimeMS`, gunicorn/Flask `timeout`, or TCP keepalive) enforces a ~10s deadline that kills requests before `curl --max-time=30s` takes effect. This explains the uniform latency ceiling across all modes.

3. **Reboot inconsistency** (medium confidence): The between-run reboot protocol may not fully reset all state. If Docker volume cleanup or OVS flow-table state persists across reboots, later runs may start in a subtly different state.

## Next Steps

1. **Investigate the bimodal failure pattern**: Compare the "healthy" and "degraded" runs within each mode to identify what differs (container events, resource utilization patterns, MongoDB slow-query logs).

2. **Identify the ~10s server-side timeout**: Audit all components in the request path (MongoDB driver `serverSelectionTimeoutMS`, Flask/gunicorn config, OVS flow-rule timeouts) for a hard 10s limit.

3. **Consider re-running degraded-mode replicates**: If the degraded runs are caused by an identifiable and fixable issue, replacing them would yield the clean dataset the plan intended.

4. **Cross-reference with v2-lite**: Compare these results with the v2-lite dataset (8 runs, same WAN=200ms, curl=10s and 30s) to determine whether the bimodal pattern is new or was present but masked by lower replicate count (n=1 per mode in v2-lite).

### What Diverged From Plan Expectations

Beyond the failure rate variance, three additional divergences occurred:

1. **Latency remains censored at ~10.5s despite curl=30s.** The plan's hypothesis #5 expected curl=30s to "uncensor the latency distribution" with Poll-30s p95 > 10.55s (cf. v2-lite). In practice, p99 is only 0.1–0.3s above p95 across ALL modes, indicating a server-side ~10s timeout bottleneck. The curl change moved the client-side cap from 10s to 30s but did not remove the server-side cap.

2. **Reaction latency metric fails under extreme degradation.** push_3 (60.6% failure) recorded 0 breach-detector reaction events despite obvious system distress. The metric requires a breach→spawn_done event pair, which may not occur if the system is so degraded that spawns either fail or complete outside the detection window.

3. **n=3 does not produce useful μ±σ when variance is bimodal.** The plan assumed within-mode variance would be small enough for clean per-mode comparisons. Instead, the variance is dominated by a binary healthy/degraded regime switch, making the mean a misleading statistic. The thesis would be better served by reporting healthy-run metrics separately and characterizing the degraded regime as a distinct operational mode.

### Why storage_storm Is Degraded — Consistent With v2-lite

storage_storm is inherently the most demanding phase — **~192 cross-region writes/second** (48 clients × 4 RPS × 90% cross-region × 50% write mix). The content_update endpoint writes directly to MongoDB's primary, generating oplog traffic that stresses all replica members. content_aggregate runs full-collection scan aggregations. At curl=30s, slow writes survive longer, accumulating MongoDB load.

v2-lite's poll30_t30 also saw storage_storm at 63.1% — nearly identical to v2final's poll30_3 at 57.7%. **This pattern is workload-inherent, not a v2final regression.** The difference is that v2-lite's single runs isolated the damage to storage_storm, while v2final's degraded runs show cascading failures into subsequent phases (reverse_hotspot, compute_spike), suggesting the MongoDB saturation from storage_storm persists into later phases when the system cannot recover in time.
